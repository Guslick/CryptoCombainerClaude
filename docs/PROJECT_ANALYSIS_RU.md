# Анализ проекта CryptoCombainer

## 1) Цель и позиционирование проекта

CryptoCombainer — это Java-приложение для алгоритмической торговли криптовалютами с двумя интерфейсами управления:
- Telegram-бот;
- Telegram MiniApp / web-панель.

По коду видно, что проект решает практическую задачу **запуска торговых сессий для разных режимов** (tester, research/backtest, Binance testnet/real), управления ими и сохранения состояния между рестартами.

Ключевая идея: дать пользователю «операционную консоль» для быстрых запусков стратегий и контроля торговли без отдельного тяжелого бэкенда/SPA-стека.

## 2) Архитектура и функционал

## 2.1 Точка входа и жизненный цикл

- `App.java` поднимает логирование, загружает список монет (локально или с CoinGecko), затем запускает MiniApp сервер и/или Telegram-бот в зависимости от конфигурации.
- Конфиг централизован в `AppConfig` (приоритет env -> `config.properties` -> default).
- Есть совместимый слой `RuntimeConfig` для legacy-ключей.

Это делает запуск гибким для локальной среды, systemd и серверного деплоя.

## 2.2 Торговые режимы и сессии

Через `TradingSessionManager` реализована модель **пер-пользовательских сессий**:
- отдельные менеджеры по userId;
- хранение сессий/состояния в user-scoped каталогах;
- live-обновления статуса (балансы, текущая цена, цели, статистика);
- журнал событий по сессии.

Это сильная часть архитектуры, потому что снимает риски взаимного влияния пользователей и готовит основу для масштабирования.

## 2.3 Стратегии

### Reversal Points
Классическая логика входа/выхода по разворотам, с параметрами:
- `buyGap`;
- `sellWithProfitGap`;
- `sellWithLossGap`.

Плюс:
- persistence состояния;
- отправка уведомлений;
- учет комиссий и статистики сделок;
- поддержка режима recapitalization.

### ATR+EMA Adaptive
Эволюция Reversal с:
- фильтром тренда EMA;
- адаптацией порогов через ATR multiplier;
- принудительным выходом при пробое EMA вниз.

И live-trader, и backtester реализованы отдельно, что облегчает эксперименты и сравнение результатов.

## 2.4 Надежность

Есть централизованные `ErrorHandler` и `RetryPolicy`, плюс документация по hardening. Также есть:
- autosave состояния;
- backup rotation;
- shutdown hooks для сохранения активных трейдеров.

Для трейдингового бота это критически важный базис.

## 2.5 UI/UX (MiniApp)

`index.html` — большой single-file frontend с мобильной ориентацией:
- вкладки/подвкладки;
- карточки сессий;
- графики;
- логи/ивенты;
- элементы для профиля и выбора биржи.

Плюс: быстро разворачивается и просто деплоится.

Минус: масштабируемость фронта и сложность сопровождения со временем.

## 3) Сильные стороны

1. **Практичная end-to-end система**: от получения монет и рыночных данных до live-торговли, бэктестов и UI-контроля.
2. **User isolation в сессиях**: зрелое решение для мультитенантности на уровне файлов и runtime-моделей.
3. **Стейтфул-устойчивость**: автосейв, восстановление, бэкапы, shutdown hooks.
4. **Наличие двух стратегических контуров**: базовый (Reversal) и адаптивный (ATR+EMA).
5. **Неплохая операционная готовность**: systemd-инструкции, deployment гайды, логирование.

## 4) Слабые стороны и риски

1. **Сильная связанность стратегии и инфраструктуры**
   - В стратегиях много инфраструктурной логики (Telegram-уведомления, графики, state persistence, session logging).
   - Это усложняет unit-тестирование и переиспользование ядра сигналов.

2. **Дублирование логики между live и backtest слоями**
   - Часть алгоритмов и расчётов повторяется в разных классах.
   - Риск рассинхронизации поведения «в тесте» и «в проде».

3. **Frontend как монолитный HTML/JS-файл**
   - Быстрый старт, но сложная эволюция: тяжело рефакторить, писать компонентные тесты, поддерживать стиль/бизнес-логику.

4. **Ограниченный охват автотестами**
   - В репозитории есть тесты аккаунтов, но мало признаков системных тестов стратегий, session manager и edge-case сценариев восстановления.

5. **Смешение доменной и операционной ответственности**
   - Например, доменная стратегия напрямую формирует пользовательские сообщения.
   - Это мешает делать чистый слой «решение стратегии -> оркестратор -> каналы уведомлений».

6. **Исторический источник данных и model risk**
   - Логика backtest ориентирована на price-series; микроструктура/проскальзывание/латентность ограниченно моделируются.
   - Результаты оптимизации могут быть смещены относительно реального исполнения.

## 5) Оценка стратегий

## 5.1 Reversal Points

**Плюсы:**
- простая интерпретируемая логика;
- легко оптимизировать параметры;
- понятная связь сигналов с локальными максимумами/минимумами.

**Минусы:**
- нет встроенного понимания тренда (без фильтра);
- чувствительность к шуму на боковом/рваном рынке;
- фиксированные пороги хуже работают при смене волатильности.

**Где уместна:**
- быстрые поисковые эксперименты;
- рынки со стабильной внутридневной структурой.

## 5.2 ATR+EMA Adaptive

**Плюсы:**
- адаптивные пороги к волатильности;
- EMA-фильтр сокращает входы против тренда;
- дополнительный защитный выход (trend break).

**Минусы:**
- больше параметров и выше риск переоптимизации;
- lag EMA может задерживать входы/выходы;
- на резких разворотах возможно «запаздывание» сигнала.

**Где уместна:**
- режимы с сильно меняющейся волатильностью;
- более консервативный стиль относительно базового Reversal.

## 6) Приоритетные улучшения

## P0 (высокий приоритет, 1-2 итерации)

1. **Выделить Strategy Core (pure domain layer)**
   - Вынести сигнальную логику в чистые классы без I/O.
   - Интерфейс типа `StrategyEngine.onPriceTick(...) -> Decision`.
   - Telegram, chart, persistence, exchange исполнение — в оркестраторе.

2. **Единый backtest/live execution contract**
   - Общий слой комиссий, правил размера позиции, stop/profit логики.
   - Цель: одинаковое поведение алгоритма в историческом и real-time режиме.

3. **Risk Manager как отдельный модуль**
   - лимит риска на сделку (% equity), дневной stop, max consecutive loss;
   - kill-switch при аномалиях API/проскальзывании;
   - guardrails для запуска real trading.

4. **Тесты на критические сценарии восстановления**
   - crash/restart в момент открытой позиции;
   - повреждённый state + восстановление из backup;
   - корректность resumed-session параметров.

## P1 (средний приоритет)

5. **Версионирование state-схемы**
   - добавить `stateVersion` и миграции для backward compatibility.

6. **Метрики и наблюдаемость**
   - structured logs + метрики (trade latency, API error rate, pnl drawdown, restore success rate).

7. **Refactor фронта в модульную структуру**
   - даже без полного перехода на framework: разбить JS/CSS/HTML на отдельные модули.

8. **Walk-forward и out-of-sample протокол оптимизации**
   - фиксированный pipeline: train window -> validation window -> forward test.

## P2 (дальнейшее развитие)

9. **Exchange abstraction 2.0**
   - расширить на несколько бирж с унифицированным order model.

10. **Портфельный режим**
   - не только 1 актив/сессия, а риск-бюджет на корзину и корреляционные ограничения.

## 7) Практический roadmap на 6 недель

- **Неделя 1-2:** выделение strategy-core + unit tests decision logic.
- **Неделя 3:** risk-manager + pre-trade validation.
- **Неделя 4:** state versioning + restore tests + chaos сценарии (forced kill).
- **Неделя 5:** walk-forward framework и отчеты (in/out sample).
- **Неделя 6:** modularization MiniApp + UX шлифовка журналов/метрик.

## 8) Итог

Проект уже выглядит как **рабочая торговая платформа индивидуального класса** с сильным упором на эксплуатацию (Telegram + MiniApp + восстановление сессий). Основной потенциал роста — в инженерной дисциплине вокруг стратегий: разделение домена и инфраструктуры, унификация backtest/live, и стандартизация риск-контроля. При реализации этих шагов качество решений и предсказуемость поведения системы вырастут заметно.
