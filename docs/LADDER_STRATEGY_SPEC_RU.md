# Стратегия «Лесенка» — спецификация (v1)

## 1) Цель
Реализовать новую торговую стратегию «Лесенка» с чистым (pure) ядром принятия решений,
чтобы одинаковая логика могла использоваться в live-торговле и в бэктесте.

## 2) Входные параметры
- `orderUsdt`: фиксированная сумма каждой покупки в USDT.
- `priceStepPercent`: шаг сетки в процентах (например `10` для 10%).

Производная константа:
- `k = 1 + priceStepPercent / 100`.

## 3) Базовые определения
- `basePrice`: первая доступная цена после запуска стратегии.
- Уровни сетки:
  - вверх: `basePrice * k^n`, где `n >= 0`
  - вниз: `basePrice / k^n`, где `n >= 0`
- `stepIndex` (целое число): индекс текущей ступени относительно `basePrice`.
  - Для `price >= basePrice`: `floor(log(price/basePrice)/log(k))`
  - Для `price < basePrice`: `-floor(log(basePrice/price)/log(k))`

## 4) Алгоритм действий
1. На первом тике фиксируется `basePrice`.
2. При переходе на новую ступень (`stepIndex != lastExecutedStepIndex`) выполняется покупка:
   - `qty = orderUsdt / currentPrice`.
   - В реестр лотов добавляется запись `{ buyPrice=currentPrice, quantity=qty }`.
3. После обработки цены проверяется самое дешёвое (по `buyPrice`) открытое плечо:
   - Если `currentPrice >= minBuyPrice * k`, выполняется продажа `quantity` этого лота.
   - Проданный лот удаляется из реестра.

## 5) Состояние стратегии
Хранится состояние:
- `initialized`
- `basePrice`
- `lastExecutedStepIndex`
- `openLots[]` (лоты покупки)

## 6) Исполнение (execution contract)
Pure-ядро выдаёт решение, фактическое исполнение делается отдельным исполнителем:
- `buy(price, usdtAmount)`
- `sell(price, quantity)`

Это позволяет подключать:
- исполнение через биржу (live)
- симулятор (backtest)
- dry-run для тестов

## 7) Ограничения версии v1
- Только фиксированная сумма покупки (`orderUsdt`), без рекапитализации.
- Одна продажа за тик (самый дешёвый лот).
- Без фильтров тренда/волатильности.

## 8) План интеграции инфраструктуры
1. Core: decision engine + orchestrator + executor contract.
2. Unit tests для переходов между уровнями и продажи лотов.
3. Подключение к backtest-режиму.
4. Подключение к live-режимам и API/UI.
5. Метрики, отчёты, визуализация сделок.

## 9) Критерии готовности этапа 1
- Есть детерминированный `LadderDecisionEngine`.
- Есть unit-тесты на ключевые сценарии.
- Есть `LadderExecutionOrchestrator` с контрактом исполнителя.
